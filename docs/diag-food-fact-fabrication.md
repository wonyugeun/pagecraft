# 진단 — 식품 카피 수치 날조 (단호박 "20Brix·강분질·신품종")

> ⚠️ 식품 당도(Brix) 날조 = 표시광고법 위반 = 출시 전 차단해야 할 법적 지뢰.
> **진단 전용. 코드 수정 없음.**

---

## 0. 결론 한 줄

universalFactGuard는 "당도 Brix"를 **명시적으로 금지**하고 전 카테고리에 적용되지만, **식품 카테고리 프롬프트가 같은 것을 "반드시 사용할 용어"로 강하게 지시**(예시 "당도 15 Brix" 포함)하고, **식품에는 화장품 같은 "가드 우선" 선언이 없어** 상충 지시 중 positive 쪽(날조)이 이긴다. "강분질·신품종"은 가드 목록 밖(품종 특성 서술)이라 LLM 일반지식으로 통과.

---

## 1. 가드는 Brix를 알고 있다 (그런데도 뚫림)

`lib/copyGuards.ts` `universalFactGuard` (전 카테고리 적용, `getCategoryCopyGuard` line 84):
```
■ 미입력 수치 생성 금지
  셀러가 입력하지 않은 구체 수치(함량%, 당도 브릭스, 중량, …)를 만들어내지 마세요.
■ 미입력 사실 생성 금지
  · 인증·등급 명칭 (HACCP, GAP …)  · 이력·추적 시스템  · 원산지 보증 …
```
→ Brix·HACCP·GAP·이력추적은 **이미 금지 목록에 있음**. 가드 자체 범위 문제는 아니다(아래 3 참조).

---

## 2. 날조 경로 — 어느 단계인가

### Stage1(strategy): 비교적 안전, 단 품종지식 유입 여지
- `lib/stages/strategy.ts`: 카테고리 프롬프트 **미사용**. `universalFactGuard` + "main_weapon은 입력 정보에 실재하는 것에서만 도출… 없는 사실 생성 금지"를 2회 명시.
- 그러나 상품명 "보우짱 밤호박"의 **품종명 자체**를 단서로, 모델이 일반지식(보우짱=일본 미니 단호박 품종, 분질감)을 main_weapon/스토리에 끌어올 여지는 남음.

### ⭐ Stage3(copy): 여기가 주 발생지
`lib/stages/copy.ts` 조립:
- **system 프롬프트** = 식품 카테고리 프롬프트(`categoryPrompts.ts` `'식품'.system`) + blogModeRules
- **user 프롬프트** = 섹션 + sectionGuide + blockSpec + **copyGuard(universalFactGuard)** + COPY_PRINCIPLES

식품 카테고리 system 프롬프트(`categoryPrompts.ts:120~139`)가 **날조를 직접 지시·예시**:
```
[반드시 사용할 업계 전문 용어]
당도(Brix) · GAP 우수농산물 인증 · HACCP · 콜드체인 · IQF · …      ← "반드시 사용"
① Attention: "황금빛 꿀처럼 흐르는 당도 15 Brix …"                  ← Brix 수치 예시
② Interest:  "당도 측정기로 직접 선별해 보내드립니다"
[차별점] "생산 이력 번호로 농장까지 추적 가능"                       ← 이력추적 예시
        "Brix OO 이하는 출하하지 않습니다"                          ← Brix 기준 예시
sectionGuide: "원산지 섹션: 생산지역+생산자명+재배방식+인증 스토리"  ← 인증·생산자 스토리 요구
```
→ system(=강한 positive 지시 + 구체 예시)이 "Brix를 반드시 써라"라고 하고, user의 가드가 "미입력 Brix 금지"라고 한다. **한 프롬프트 안의 정면 충돌.** 모델은 구체 예시가 붙은 positive 지시를 따라 그럴듯한 값(20Brix)을 발명.

### "강분질·신품종" = LLM 일반지식
- 식감 형용("강분질")·"신품종"은 수치도 인증도 이력도 아니라 **가드 명시 목록 밖**.
- 식품 sectionGuide가 "품종·재배방식 스토리"를 요구 → 모델이 보우짱 품종 일반지식을 서술로 끌어옴. 가드가 못 잡음.

## 3. universalFactGuard가 식품에서 왜 안 막았나 (2가지 한계)

1. **우선순위 미설정(핵심).**
   화장품 가드(`cosmeticsCopyGuard` line 33)에는 명시적 선언이 있다:
   > "※ 이 가드는 [반드시 사용할 업계 전문 용어]에 TEWL·분자량(Da)… 있어도, 본 가드에 해당하면 절대 사용하지 말고 [허용 표현]으로 우회하세요."

   즉 화장품은 "카테고리 반드시-사용 목록보다 가드가 우선"이라고 못박는다. **식품에는 이 선언이 없다** — `COPY_GUARDS`에 `'화장품'`만 있고, 식품은 `getCategoryCopyGuard`가 universalFactGuard만 반환(우선순위 문구 0). 그래서 식품 system의 "반드시 사용 Brix"를 이길 장치가 없다.

2. **가드 범위 밖 항목.**
   가드는 수치·인증·이력·수상·시험기관·원산지보증은 잡지만, **품종 특성·식감 서술("강분질", "신품종", "밤처럼 포슬포슬")** 같은 정성적 사실 주장은 목록에 없어 통과.

추가 구조 문제: **카테고리 프롬프트 예시 자체가 날조 템플릿**이다("15 Brix", "생산 이력 번호로 추적", "Brix OO 이하 미출하"). 셀러 미입력이어도 모델이 예시 형태를 그대로 채워 넣게 유도.

---

## 4. 다른 미입력 사실 점검 (식품 카테고리 프롬프트가 유도하는 날조군)

단호박 외에도 식품 전반에서 다음이 셀러 미입력 시 자동 생성될 구조적 위험:
- **수치**: 당도 Brix, "100g당 단백질 28g" 류 영양 수치(예시 line 131), 산도(%), 수분함량, 정형 중량
- **인증**: GAP, HACCP (system "반드시 사용" 목록)
- **이력/유통**: "생산 이력 번호로 농장까지 추적", 콜드체인, IQF, 산지 직발송 보증
- **생산자**: "30년 외길 농부 OOO", "생산자 OOO 실명" (예시가 실명 생성을 유도)
- **품종/식감**: 강분질, 신품종, 분질/점질 등 품종 일반지식 서술
- **urgency**: "이번 시즌 수확량의 단 30%" 류 미검증 한정 수량

※ 본 진단 시점에 그 단호박 생성 결과(셀러 입력 원문 + 카피 전문)는 저장소에 persist되지 않아 1:1 대조 출력은 불가. 위는 코드(프롬프트) 기반 발생 경로 분석. 정확한 1:1 목록이 필요하면 동일 입력으로 재생성해 대조 가능(별도 요청).

---

## 5. 권장 수정 방향 (전 카테고리 미입력 수치/사실 가드 강화)

1. **universalFactGuard에 "카테고리 반드시-사용 목록보다 우선" 선언 추가** (화장품 가드 line 33과 동형).
   → "system의 [반드시 사용할 업계 전문 용어]에 Brix·인증·이력이 있어도, 셀러 미입력이면 본 가드가 우선해 사용 금지."
2. **식품 카테고리 프롬프트의 날조-유도 예시 무해화.**
   - "반드시 사용할 업계 전문 용어"의 Brix·GAP·HACCP·이력추적을 "**셀러가 입력한 경우에만** 사용할 용어"로 조건화.
   - AIDA/차별점 예시의 구체 수치·실명·이력번호를 플레이스홀더 + "셀러 입력 시에만" 단서로 교체.
3. **가드 목록에 품종/식감 정성 주장 추가**: "품종 특성·식감·신품종 여부 등도 셀러 미입력이면 단정하지 말 것(상품명에 품종명이 있어도 특성은 일반지식으로 서술 금지)."
4. **식품 전용 factGuard 신설**(현재 화장품만 있음): Brix·영양수치·인증·이력·생산자실명·한정수량을 미입력 시 금지로 명문화.
5. (선택) **다운스트림 정규식 검출**: "\d+\s?Brix", "HACCP|GAP", "생산 이력" 등을 셀러 입력에 없으면 경고/차단하는 후처리 안전망.

> 핵심: 가드는 이미 Brix를 알지만 **식품 카테고리 프롬프트의 "반드시 사용" 지시를 이길 우선순위가 없어서** 뚫렸다. 1·2번(우선순위 선언 + 카테고리 예시 조건화)이 1순위.
