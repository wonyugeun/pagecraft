# ResultScreen 디자인 시스템 — "문서 → 상세페이지"

> 설계 문서. **코드 수정 없음.** Stage1·2·3·4 변경 없음.
> 목적: 결과물을 "흰 배경 + 제목 + 본문 반복(문서)"에서 "강조↓설명↓강조↓설명(상세페이지)"으로 바꾸는 **디자인 시스템 정의**.
> 전제: Stage1 전략·Stage2 구조·Stage3 카피·Stage4 V2(image_mission) 전부 정상. 카피 소실(subcopy 0/14)은 직전 작업에서 차단 완료. 병목은 **ResultScreen 렌더 디자인**.
> ⚠️ 시안 없는 블록의 색상·여백·폰트·타이포는 **절대 임의 디자인 금지** — "GPT 시안 필요"로 표기.

---

## 0. 두 종류의 "블록"을 구분하라 (혼동 주의)

| 층위 | 무엇 | 예 |
|---|---|---|
| **구조 블록**(Stage3 출력) | 카피가 만든 데이터 단위 | `heading·paragraph·checklist·steps·iconcards·stats·compare·quote·faq·cta·image` |
| **디자인 블록**(이 문서의 9종) | 화면에서 "한 섹션을 어떤 시각 언어로 보여줄지" | `Hero·Problem·Cause·Feature·KPI·Quote·FAQ·Comparison·CTA` |

→ 디자인 블록은 **섹션 단위 시각 언어**, 구조 블록은 그 안에 담기는 **재료**. 매핑은 §4.
현재 코드 위치: `components/result/BlockRenderer.tsx`(구조 블록 렌더), `components/screens/ResultScreen.tsx`(BlogSection 조립).

## 1. 디자인 블록 9종

> 시안 상태: **Hero / Comparison / CTA = 적용된 GPT 시안 있음**(현재 구현 기준 기술) / **Problem · Cause · Feature · KPI · Quote · FAQ = 시안 없음(GPT 시안 필요, 디자인 임의 금지)**.

### 1) Hero — ✅ 시안 있음
- **역할**: 첫 화면. 이 제품이 무엇이고 왜 봐야 하는지 3초 안에 각인.
- **감정 목표**: "어, 이거 내 얘기 / 좋아 보인다" — 신뢰 + 끌림.
- **사용 조건**: 첫 섹션(`isFirst`) 전용. 1페이지 1개.
- **모바일 규칙**: 패딩 축소(px-8 py-12), headline 36px(데스크탑 58px), KPI 3열 유지(좁으면 값/라벨 축소).
- **디자인 규칙(현재 GPT 시안)**: `max-w-760 · rounded-36 카드 · soft→white 세로 그라데이션 · "추천 상품" 배지(Sparkles)` / headline `36→58px extrabold tracking-[-0.04em]` / subcopy `16→18px zinc-600` / KPI `grid-cols-3, value primary extrabold 22→30px + label zinc-500` / 제품 이미지 영역 `rounded-30 흰 카드`(이미지 없으면 placeholder). 색은 제품 테마(primary/accent/soft/softBorder).

### 2) Problem (공감) — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 타깃의 불편/고민을 "내 얘기"로 비추어 공감 회로 열기.
- **감정 목표**: "맞아, 딱 내 상황이야."
- **사용 조건**: emotion_goal=공감 섹션. 보통 2번째.
- **모바일 규칙(요구)**: 상황 이미지(Stage4 V2 공감컷=붉어진 피부)와 공감 문장이 세로로 자연스럽게. 구체 치수는 시안.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.** 색·여백·폰트 임의 금지. (의도: 설명문이 아니라 "공감 장면"으로 보여야 함 — 본문 나열 금지.)

### 3) Cause (원인) — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 문제의 진짜 원인을 논리적으로 짚어 "해결이 필요하다" 납득.
- **감정 목표**: "아, 그래서 그랬구나."
- **사용 조건**: emotion_goal=원인 섹션.
- **모바일 규칙(요구)**: 원인 단계(steps)가 세로로 끊겨 읽히게.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.** (의도: 인과가 한눈에 — before/after 또는 단계적 시각화. 임의 디자인 금지.)

### 4) Feature (솔루션) — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 제품이 문제를 어떻게 푸는지(성분·방식·특징) 제시.
- **감정 목표**: "이거면 되겠다."
- **사용 조건**: emotion_goal=솔루션 섹션. iconcards/steps 동반 多.
- **모바일 규칙(요구)**: 성분 카드 2~3열 → 1열, 제품+성분 이미지 동반.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.** (의도: 카드 나열이 아니라 "해결책 제시"로 보여야 함.)

### 5) KPI (숫자 강조) — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 핵심 수치(셀러 입력값만)를 시각적으로 때려, 신뢰·임팩트.
- **감정 목표**: "오, 숫자가 확실하네."
- **사용 조건**: 신뢰/스펙 섹션, `stats` 구조 블록이 있을 때. ⚠️ 셀러 미입력 수치 생성 금지(가드 준수).
- **모바일 규칙(요구)**: 숫자 크기 유지(작아지면 임팩트 소실), 2~3열.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.** (의도: "숫자가 먼저, 의미가 다음." 현재 StatsBlock은 임시 — 시안으로 대체 전제.)

### 6) Quote (후기) — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 제3자 목소리로 신뢰 증폭. ⚠️ 셀러 미입력 시 가짜 후기 금지(기대 시나리오로 처리).
- **감정 목표**: "남들도 효과 봤구나."
- **사용 조건**: 신뢰 섹션 중 `quote` 블록.
- **모바일 규칙(요구)**: 인용문 가독성 우선.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.**

### 7) FAQ — ⚠️ 시안 없음 (GPT 시안 필요)
- **역할**: 구매 직전 남은 의심 선제 해소.
- **감정 목표**: "그 걱정도 해결되네."
- **사용 조건**: 구매장벽/Q&A 섹션, `faq` 블록.
- **모바일 규칙(요구)**: 아코디언/리스트 탭 영역 충분히.
- **디자인 규칙**: ⚠️ **GPT 시안 필요.**

### 8) Comparison — ✅ 시안 있음
- **역할**: 경쟁/대안 대비 이 제품의 우위를 한눈에.
- **감정 목표**: "확실히 이게 낫네."
- **사용 조건**: 비교/차별화 섹션, `compare` 블록.
- **모바일 규칙(현재 시안)**: 데스크탑 2열(`1fr 1fr`) → 모바일 1열, 추천 카드 scale 강조는 모바일에서 해제.
- **디자인 규칙(현재 GPT 시안)**: 2카드(일반 vs 우리). **우리 카드 강조** = `border 2px primary · soft 배경 · scale 1.03 · shadow · "추천" 배지(top:-14)`. 일반 카드 = `opacity 0.75 · 얇은 보더`. 행 = 항목 라벨 + 값(우리 값은 primary·700).

### 9) CTA — ✅ 시안 있음
- **역할**: 마지막 구매 결정 유도.
- **감정 목표**: "지금 사야겠다."
- **사용 조건**: 마지막 섹션. emotion_goal=구매.
- **모바일 규칙(현재 시안)**: 패딩 56→36, headline 42→30, 버튼 width 320 → 100%.
- **디자인 규칙(현재 GPT 시안)**: `max-w-760 · rounded-36 · soft 배경 중앙정렬` / headline `42(30)px extrabold` / 버튼 `height 60 · primary 배경 · 흰 글씨 · ArrowRight · radius 18`.

## 2. 섹션 리듬 정의 (핵심)

### 현재 (문제)
```
[흰배경 제목+본문+카드] → [흰배경 제목+본문+카드] → … (전부 동일 밀도) = 문서
```

### 목표 (상세페이지)
```
강조(Hero) ↓ 설명(Problem) ↓ 설명(Cause) ↓ 강조(Feature 일부) ↓ 설명 ↓ 숫자강조(KPI) ↓ 신뢰(Comparison/Quote) ↓ 설명(FAQ) ↓ 강조(CTA)
```
= **강조와 설명이 번갈아** 나오며 시선이 쉬었다 집중했다 반복.

### 강조 섹션 vs 설명 섹션
| 분류 | 블록 | 시각 특성(의도) |
|---|---|---|
| **강조 섹션** | Hero · KPI · Comparison · CTA | 배경/색/스케일로 튀어야. 페이지의 "쉼표·느낌표" |
| **설명 섹션** | Problem · Cause · Feature · Quote · FAQ | 차분히 읽히게. 강조 사이를 잇는 "본문" |

→ 강조 섹션이 너무 잦으면 피로, 설명만 이어지면 문서. **강조 1 : 설명 1~2** 간격이 기준.

### 섹션 role(emotion_goal) → 디자인 블록 매핑
| Stage 의미(emotion_goal) | 디자인 블록 |
|---|---|
| 첫 화면 | **Hero** |
| 공감 | **Problem** |
| 원인 | **Cause** |
| 솔루션 | **Feature** |
| 신뢰 | **Comparison · KPI · Quote** (재료에 따라) |
| 구매 | **CTA** |

### 페이지 전체 리듬 (예시 골격)
```
Hero(강조) → Problem(설명) → Cause(설명) → Feature(설명, 일부 강조) →
KPI(강조) → Comparison(강조) → Quote(설명) → FAQ(설명) → CTA(강조)
```
강조(Hero)─설명─설명─설명─**강조(KPI)**─**강조(Comparison)**─설명─설명─**강조(CTA)**.
※ 신뢰 구간(KPI+Comparison)이 강조 2연속이면 사이에 Quote(설명) 끼워 리듬 회복.

## 3. 현재 화면 문제 정리

| # | 문제 | 원인(가설) |

## 4. 블록 ↔ 섹션 매핑 규칙

Stage3 구조 블록 출력이 9개 디자인 블록으로 연결되는 규칙:

| Stage3 구조 블록 | → 디자인 블록 | 비고 |
|---|---|---|
| (첫 섹션 위치) | **Hero** | 블록 타입 아님, `isFirst`로 판정(현재 그대로) |
| `checklist` | **Problem** | 공감 체크리스트 = "내 상황" 목록 |
| `steps` | **Cause** 또는 **Feature** | 원인 단계 → Cause / 사용·해결 단계 → Feature |
| `iconcards` | **Feature** | 성분·특징 카드 |
| `paragraph`/`heading` | (설명 본문/구분선) | 강조 아님. 섹션 본문·소제목 보조 |
| `stats` | **KPI** | 숫자 강조 |
| `compare` | **Comparison** | ✅ 시안 |
| `quote` | **Quote** | 후기 |
| `faq` | **FAQ** | Q&A |
| `cta` | **CTA** | ✅ 시안 |
| `image` | (블록 내 이미지) | Stage4 V2 브리프→Gemini 배선은 별도 작업 |

**우선순위 규칙**: 한 섹션이 여러 신호를 가지면 **emotion_goal(섹션 역할) > 구조 블록 타입** 순으로 디자인 블록 결정.
예: 신뢰 섹션에 `stats`+`compare` 동시 → KPI와 Comparison 둘 다 쓰되 §2 리듬(강조 연속 회피)으로 배치.

## 5. GPT 시안 요청 목록 (시안 없는 6블록)

각 블록마다 GPT에 아래 5종을 요청한다. **공통 제약**: 제품 테마 토큰(primary/accent/soft/softBorder) 사용 / 모바일 우선 / 카피가 디자인에 묻히지 않을 것 / Tailwind + React.

| 블록 | 요청 항목 (레이아웃 · 타이포 · 모바일 · 색상규칙 · React/Tailwind 코드) |
|---|---|
| **Problem** | 공감 장면 레이아웃(공감문장+상황이미지+체크리스트), 공감문장 타이포 위계, 모바일 세로 흐름, 테마색 사용 범위, 코드 |
| **Cause** | 인과 시각화(단계/before-after) 레이아웃, 단계 타이포, 모바일 단계 스택, 색상, 코드 |
| **Feature** | 성분/특징 카드 레이아웃(제품+성분 동반), 카드 타이포, 모바일 1열, 색, 코드 |
| **KPI** | 숫자 강조 레이아웃(값>의미>설명), 숫자 타이포 스케일, 모바일 숫자 유지 규칙, 강조색, 코드 |
| **Quote** | 후기 카드 레이아웃(가짜후기 방지 톤), 인용 타이포, 모바일, 색, 코드 |
| **FAQ** | 아코디언/리스트 레이아웃, Q/A 타이포, 모바일 탭 영역, 색, 코드 |

> 요청 시 **§6 시각 우선순위**를 각 블록 스펙으로 함께 전달 — 시안이 위계를 반영하도록.

## 6. 블록별 시각 우선순위 (유근님 확정안)

> ⚠️ 이 항목 없으면 구현 후에도 "카피 안 읽힘"이 재발. 각 블록의 **1순위 → 2순위 → 보조**.

- **Hero**: ①headline ②제품 이미지 ③subcopy ④KPI
  - ⭐단 제품 이미지 없을 때: ①headline ②subcopy ③KPI ④placeholder(작게)
  - — 이유: placeholder가 실제 제품보다 존재감 크면 안 됨(Hero 실패 원인)
- **Problem(공감)**: ①공감 문장 ②상황 이미지 ③설명 — 설명 아니라 "내 얘기다"가 먼저
- **Cause(원인)**: ①원인 메시지 ②원인 시각화 ③설명 — 원인 이해가 먼저
- **Feature(솔루션)**: ①해결 메시지 ②성분/기능 카드 ③설명
  - ⭐중요: 성분 자체보다 "그래서 내 문제를 어떻게 해결하는가"가 먼저
- **KPI**: ①숫자 ②의미 ③설명
- **Quote**: ①후기 핵심 문장 ②후기 본문 ③작성자 정보 (가짜 후기 금지 유지)
- **FAQ**: ①질문 ②답변
- **Comparison**: ①차이점 ②우리 제품 ③일반 제품
  - ⭐중요: 사용자는 "뭐가 다른데?"를 보러 옴. 좌우 카드 동등 비중 금지
- **CTA**: ①구매 메시지 ②CTA 버튼(거의 동급) ③보조 설명
  - — CTA는 버튼 안 보이면 의미 없음

**원칙**: body는 항상 보조(1순위 절대 아님). headline/핵심메시지가 먼저.
