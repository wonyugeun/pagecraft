## 0. 한 줄 결론

**병목은 모델도, 브리프 텍스트 품질도 아니라 "구조"다.** ① 제품 일관성 = 레퍼런스 배선 문제(B에서 완전 해결됨), ② 카피-이미지 내용 정합 = "모든 컷을 제품 정물로 수렴시키는 shot 규칙 + 인물 금지 + body 미전달"이라는 구조적 제약.

## 2. 6개 평가 항목 × 섹션 판정

| 항목 | ①공감 | ②원인 | ③솔루션 | ④신뢰 | ⑤CTA |
|---|---|---|---|---|---|
| **1. 이미지-카피 정합성** | △ 분위기만 일치, 내용(빨개짐) 없음 | ✕ 원인(향/색소/끈적) 시각화 실패, 오독 위험 | △ subtraction 은유 미전달 | △ clinical 분위기만, 근거 약함 | △ 마감 분위기만, 행동요소 없음 |
| **2. 섹션 역할 분화** | ◯ 5컷의 구도·소품·무드는 서로 다름 (욕실/대조/자갈/클리닉/골든). 단 **전부 제품 정물**이라는 한 틀 안에서의 분화 |
| **3. 제품 일관성 (최重)** | **A: ✕** 매 컷 다른 브랜드(SOOM·PURE NATURE·NUMBUZIN·KOREAN BIO LAB·무명) / **B: ◯** 전 컷 동일 ACUA 센텔라 토너 |
| **4. 텍스처 정확성** | A: ✕ 색·점성 임의(노란액·하늘색 등, 무색 워터리와 불일치) / B: ◯ 레퍼런스 따라 보수적(투명) |
| **5. 성분 정확성** | ◯ 병풀(둥근 centella 잎) 일관 정확. 다른 원료 오인 없음 (②의 시든 꽃잎만 은유 소품) |
| **6. 블로그형 적합성** | ◯ 카피·CTA·가격 오버레이 없음, 광고배너 아님, 본문 보조. (단 가상 영문 라벨 자동생성은 사실오류 리스크) |

## 3. ⭐ A vs B — 레퍼런스 배선이 일관성을 해결하는가? → **YES, 완전히**

- **A (레퍼런스 없음):** 5섹션이 각각 다른 브랜드·다른 병·다른 액체 색. 한 페이지에 5개 제품이 섞인 꼴. 텍스처도 임의(노랑/하늘색).
- **B (P1-0 단독컷을 셀러 업로드로 주입):** 5섹션 전부 **동일한 ACUA 센텔라 CALMING TONER 150ml**(프로스티드 병 + 세이지 라벨)로 통일. 색·형태·라벨 일관. 텍스처도 레퍼런스 따라 보수적.
- 코드 근거: `generate-image` route의 `PRODUCT_RULES`/`COMPONENT_RULES`는 `hasRefImages`일 때만 활성 → 레퍼런스 없으면 일관성 규칙이 사실상 작동 안 함. **레퍼런스 주입(productImages)이 제품 일관성의 결정적 스위치.**

## 4. 7번 — 정합성의 구조적 출처 (Stage4가 카피를 보고 이미지를 만드나?)

**보긴 본다. 그런데 "제품 정물"로만 출력한다.**

- 코드(`lib/stages/imagebrief.ts`): Stage4는 섹션별로 `name`+`role`+`mission`+`headline`(맥락, "이미지에 텍스트로 넣지 말 것")을 입력받음.
- 증거: 브리프가 섹션마다 실제로 다르게 나옴(공감=욕실 연출, 원인=대조, 솔루션=자갈 은유…). → **mission/카피를 무시하고 일반 컷을 찍는 게 아니다.** 카피를 읽고 분기는 한다.
- 그럼에도 정합성이 약한 **3대 구조적 원인:**
  1. **shot 어휘가 product-centric로 고정** — `[브리프 작성 지침]`이 shot_type을 "제품컷/연출컷/원료컷/텍스처/사용장면"으로 못박고, 화장품 비주얼 가이드가 모든 섹션 역할을 제품 중심 정물로 매핑. → 공감·원인처럼 '상황/문제'를 보여줘야 할 섹션도 제품 정물 + 은유 소품으로 번역됨.
  2. **인물·얼굴 전면 금지** — '빨개진 볼', '조마조마한 사람'(공감/원인의 핵심)을 그릴 수단 자체가 없음. 그래서 분위기(욕실·아침빛)로 우회 → 내용 불일치.
  3. **body/subcopy 미전달** — Stage4는 `headline` 한 줄만 받음. 본문 맥락 없이 추상 헤드라인에서 은유로 점프(자갈=절제, 시든잎=자극) → 의도 안 읽히거나 오독.

→ 즉 "카피-이미지 정합성" 실패는 **브리프가 카피를 안 봐서가 아니라**, 출력을 제품 정물로 수렴시키는 규칙·인물금지·입력누락 때문. 이건 브리프 "심화(문구 다듬기)"가 아니라 **브리프 입력/샷 규칙의 구조 재설계** 영역.

## 5. 결론 — 병목은 브리프냐 모델이냐 구조냐? → **구조**

- **모델(Gemini): 병목 아님.** 시키는 연출을 충실히, 고품질로 렌더.
- **브리프 텍스트 품질: 병목 아님.** 카피·mission을 읽고 섹션별로 다른 브리프를 만든다(작동 중).
- **구조가 병목 — 두 갈래:**
  1. **제품 일관성 = 레퍼런스 배선** → B에서 완전 해결 확인. **즉시 적용 가능한 1순위**(productImages를 파이프라인 전 섹션에 주입).
  2. **카피-이미지 내용 정합 = 브리프 입력/샷 규칙 구조** → ① product-centric shot 강제 완화, ② body 전달, ③ 인물금지의 대안(상황·오브젝트로 감정 표현) 설계 필요. (이번 범위 밖, 후속 과제)
